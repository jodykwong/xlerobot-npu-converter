## 📋 文档审查流程

### 审查频率

| 审查类型 | 频率 | 负责人员 | 触发条件 |
|----------|------|----------|----------|
| **代码审查伴随审查** | 每次PR | 代码审查者 | 涉及文档的代码变更 |
| **快速审查** | 每周 | 文档管理员 | 无特定触发，常规检查 |
| **深度审查** | 每月 | 文档审查小组 | 月度审查周期 |
| **季度审查** | 每季度 | 项目经理+技术负责人 | 季度结束 |
| **版本审查** | 每次版本发布 | 全团队 | 发布前检查 |

### 审查角色

- **文档管理员**: 负责文档日常维护和快速审查
- **代码审查者**: 在代码审查时同步审查相关文档
- **文档审查小组**: 3-5人团队，负责深度审查
- **项目经理**: 负责季度审查和版本审查
- **技术负责人**: 负责技术文档的准确性审查

## ✅ 代码审查伴随文档审查清单

### PR提交时检查

当提交包含文档变更的PR时，需要检查以下内容:

#### 内容完整性
- [ ] **README.md**: 项目状态是否与BMM工作流状态一致
- [ ] **相关文档**: 代码变更是否已反映在文档中
- [ ] **API文档**: 新增/修改的API是否已记录
- [ ] **CLI文档**: 命令变更是否已更新
- [ ] **示例代码**: 提供的示例是否有效且最新

#### 准确性验证
- [ ] **技术细节**: 描述的技术实现是否准确
- [ ] **版本信息**: 版本号和依赖是否正确
- [ ] **链接有效性**: 所有内部链接是否有效
- [ ] **引用准确性**: 文档间引用是否一致

#### 格式规范
- [ ] **Markdown格式**: 符合项目Markdown规范
- [ ] **代码高亮**: 代码块使用了正确的语言标识
- [ ] **表格格式**: 表格格式正确且对齐
- [ ] **图片引用**: 图片存在且路径正确

#### 新增文档检查
- [ ] **文档必要性**: 新文档是否有存在的必要
- [ ] **位置合理**: 文档放在了正确的目录
- [ ] **命名规范**: 文件命名符合规范
- [ ] **索引更新**: 是否需要更新索引或目录

### PR审查者检查项

作为PR审查者，需要验证:

#### 文档质量
- [ ] 文档内容清晰易懂
- [ ] 技术描述准确无误

#### 文档同步
- [ ] 文档与代码实现一致
